Environment Overview

Alianza provides four separate API environments to support the full software development lifecycle. Each environment is isolated with its own data, allowing you to test safely without affecting production systems.

Environment Progression

┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ Development  │───▶│      QA      │───▶│     Beta     │───▶│  Production  │
│   (D2)       │    │     (Q2)     │    │     (B2)     │    │              │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
     Testing            Validation         Pre-release          Live System
  Initial Dev          QA Testing        Beta Testing        Customer Traffic
Environment Base URL Purpose Data Persistence
Development https://api.d2.alianza.com Initial development and testing May be reset periodically
QA https://api.q2.alianza.com Quality assurance and validation Stable for testing cycles
Beta https://api.b2.alianza.com Pre-production validation Production-like stability
Production https://api.alianza.com Live customer operations Permanent, critical data

Development Environment

Base URL: https://api.d2.alianza.com

Purpose

The Development environment is designed for initial integration work, experimentation, and rapid iteration. This is where you'll spend most of your time during the development phase.

Characteristics

  • Frequent Updates: New API features are deployed here first
  • Data Resets: Data may be periodically reset or cleaned up
  • Relaxed Limits: Some rate limits may be higher for testing
  • Latest Features: Access to newest endpoints and functionality

When to Use

  • Initial API integration and SDK development
  • Testing new features and endpoints
  • Debugging and troubleshooting issues
  • Learning the API without consequences
  • Automated integration tests in CI/CD pipelines

QA Environment

Base URL: https://api.q2.alianza.com

Purpose

The QA environment provides a stable testing platform for quality assurance teams to validate integrations before beta or production deployment.

Characteristics

  • Stable Data: Data persists between testing cycles
  • Controlled Updates: API changes are more controlled
  • Realistic Limits: Rate limits match production settings
  • Testing Focus: Optimized for thorough QA testing

When to Use

  • End-to-end integration testing
  • Performance and load testing
  • User acceptance testing (UAT)
  • Regression testing before releases
  • Training and demonstration purposes

Testing Workflow

# test_integration.py
import pytest
from alianza_client import AlianzaClient

@pytest.fixture
def qa_client():
    """QA environment client for testing"""
    return AlianzaClient(
        base_url='https://api.q2.alianza.com',
        username=os.getenv('QA_USERNAME'),
        password=os.getenv('QA_PASSWORD')
    )

def test_account_creation_flow(qa_client):
    """Test complete account creation workflow"""
    # Create account
    account = qa_client.create_account({
        'accountNumber': 'QA-TEST-001',
        'name': 'QA Test Account',
        'timeZone': 'America/New_York'
    })

    assert account['accountId'] is not None

    # Create user
    user = qa_client.create_user(account['accountId'], {
        'firstName': 'Test',
        'lastName': 'User',
        'email': 'testuser@qa.example.com'
    })

    assert user['userId'] is not None

    # Cleanup
    qa_client.delete_user(user['userId'])
    qa_client.delete_account(account['accountId'])

Beta Environment

Base URL: https://api.b2.alianza.com

Purpose

The Beta environment is a production-like environment for final validation before deploying to production. It mirrors production configurations and stability.

Characteristics

  • Production Parity: Configuration matches production closely
  • Stable Environment: High availability and reliability
  • Production Limits: Same rate limits as production
  • Final Validation: Last checkpoint before production

When to Use

  • Final pre-production validation
  • Production deployment rehearsal
  • Beta customer testing
  • Performance benchmarking
  • Production-like integration testing

Deployment Checklist

# deployment_checklist.py
def validate_beta_deployment():
    """Pre-production validation checklist"""

    checklist = {
        'Authentication': validate_authentication(),
        'Account Creation': test_account_creation(),
        'User Provisioning': test_user_provisioning(),
        'Phone Number Assignment': test_phone_assignment(),
        'Call Routing': test_call_routing(),
        'Error Handling': test_error_scenarios(),
        'Rate Limiting': test_rate_limits(),
        'Concurrent Operations': test_concurrency()
    }

    # Generate report
    passed = sum(1 for result in checklist.values() if result)
    total = len(checklist)

    print(f"Beta Validation: {passed}/{total} tests passed")

    for test, result in checklist.items():
        status = "✓ PASS" if result else "✗ FAIL"
        print(f"{status} - {test}")

    return passed == total

Production Environment

Base URL: https://api.alianza.com
Critical: This is the live production environment. All operations affect real customer data and services.

Purpose

The Production environment handles live customer traffic and real business operations. This environment must be treated with extreme care and follows strict change control processes.

Characteristics

  • High Availability: 99.9%+ uptime SLA
  • Real Data: All data is permanent and business-critical
  • Strict Limits: Rate limits enforced to protect system
  • Monitoring: Comprehensive logging and alerting

Production Best Practices

DO

  • Test thoroughly in Dev → QA → Beta first
  • Use separate production credentials
  • Implement comprehensive error handling
  • Monitor all production operations
  • Use idempotent operations
  • Implement retry logic with backoff
  • Log all errors with request IDs
  • Use circuit breakers for failures

DON'T

  • Test experimental code in production
  • Share production credentials
  • Skip error handling
  • Ignore monitoring alerts
  • Make bulk changes without testing
  • Hard-code production credentials
  • Deploy without validation
  • Ignore rate limits

Production Configuration Example

# production_config.py
import os
import logging
from alianza_client import AlianzaClient

# Configure production logging
logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s [%(levelname)s] %(message)s',
    handlers=[
        logging.FileHandler('/var/log/alianza_production.log'),
        logging.StreamHandler()
    ]
)

class ProductionConfig:
    """Production environment configuration"""

    BASE_URL = 'https://api.alianza.com'
    USERNAME = os.getenv('ALIANZA_PROD_USERNAME')
    PASSWORD = os.getenv('ALIANZA_PROD_PASSWORD')
    PARTITION_ID = os.getenv('ALIANZA_PROD_PARTITION_ID')

    # Production settings
    TIMEOUT = 30
    MAX_RETRIES = 3
    BACKOFF_FACTOR = 2
    VERIFY_SSL = True

    # Monitoring
    ENABLE_METRICS = True
    ALERT_ON_ERRORS = True

# Initialize production client
prod_client = AlianzaClient(
    base_url=ProductionConfig.BASE_URL,
    username=ProductionConfig.USERNAME,
    password=ProductionConfig.PASSWORD,
    timeout=ProductionConfig.TIMEOUT,
    max_retries=ProductionConfig.MAX_RETRIES
)

# Verify connection
try:
    prod_client.authenticate()
    logging.info("Production client authenticated successfully")
except Exception as e:
    logging.error(f"Production authentication failed: {e}")
    raise

Environment Configuration

Environment-Aware Configuration

Use environment variables and configuration files to manage different environments without code changes.

# config.py
import os

class Config:
    """Base configuration"""
    ALIANZA_TIMEOUT = 30
    ALIANZA_MAX_RETRIES = 3

class DevelopmentConfig(Config):
    """Development environment"""
    ALIANZA_BASE_URL = 'https://api.d2.alianza.com'
    DEBUG = True

class QAConfig(Config):
    """QA environment"""
    ALIANZA_BASE_URL = 'https://api.q2.alianza.com'
    DEBUG = True

class BetaConfig(Config):
    """Beta environment"""
    ALIANZA_BASE_URL = 'https://api.b2.alianza.com'
    DEBUG = False

class ProductionConfig(Config):
    """Production environment"""
    ALIANZA_BASE_URL = 'https://api.alianza.com'
    DEBUG = False
    ALIANZA_TIMEOUT = 60

# Select config based on environment variable
config_map = {
    'development': DevelopmentConfig,
    'qa': QAConfig,
    'beta': BetaConfig,
    'production': ProductionConfig
}

env = os.getenv('ALIANZA_ENV', 'development')
current_config = config_map[env]()

Automated Testing Pipeline

# .github/workflows/api-tests.yml
name: API Integration Tests

on: [push, pull_request]

jobs:
  test-dev:
    name: Test Development Environment
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Run Integration Tests
        env:
          ALIANZA_BASE_URL: https://api.d2.alianza.com
          ALIANZA_USERNAME: ${{ secrets.DEV_USERNAME }}
          ALIANZA_PASSWORD: ${{ secrets.DEV_PASSWORD }}
        run: pytest tests/integration/

  test-qa:
    name: Test QA Environment
    needs: test-dev
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Run E2E Tests
        env:
          ALIANZA_BASE_URL: https://api.q2.alianza.com
          ALIANZA_USERNAME: ${{ secrets.QA_USERNAME }}
          ALIANZA_PASSWORD: ${{ secrets.QA_PASSWORD }}
        run: pytest tests/e2e/

Best Practices

Environment Management Guidelines

  • Always start in Development - Never develop against production
  • Use environment variables - Never hardcode URLs or credentials
  • Test progressively - Dev → QA → Beta → Production
  • Separate credentials - Different credentials for each environment
  • Monitor all environments - Track errors and performance
  • Document environment differences - Track any configuration variations
  • Implement health checks - Verify connectivity before operations
  • Use feature flags - Control feature rollout across environments
